BatchVault Website repository and image workflow plan

Created: 2026-06-03. Scope: D:\Projects\Code\Personal\BatchVault\WebSite.

Goal Turn the current landing project into a standalone repository named BatchVault_Website and add a GitHub Actions workflow that builds the frontend first, then builds a Docker image from the already-built frontend artifact.

Current Behavior

  • WebSite is not currently a git repository. git status reports that it is outside a git repo.
  • The sibling BatchVault naming pattern already includes BatchVault_Back, BatchVault_Front, and BatchVault_iOS.
  • The app is a Vite React frontend with a small Fastify API/server and SQLite lead storage.
  • vite.config.ts builds frontend assets into dist/client.
  • The current Dockerfile is multi-stage and runs npm run build inside Docker, then installs production dependencies in the runtime stage.
  • docker-compose.dev.yml uses node:22-bookworm-slim directly for hot reload and does not depend on the production Dockerfile.
  • There are local-only artifacts that should not enter the repository: node_modules, .data, dist, Playwright output, and temporary screenshots.

Desired Behavior

  • Initialize this folder as a standalone git repository with the remote/repository name BatchVault_Website.
  • Keep the deployable image free from frontend build-time node_modules and dev dependencies.
  • Use a two-job GitHub Actions pipeline:
    1. build_frontend: install dependencies, run npm run check, run npm run build, upload dist/client as an artifact.
    2. build_image: download the built frontend artifact, build a runtime Docker image, and push it to GHCR.
  • Publish image tags suitable for deployment, for example latest on the default branch and SHA tags for traceability.
  • Keep dev compose hot reload unchanged.

  • The GitHub owner remains dzarlax.
  • The default branch should be main unless GitHub creates a different default.
  • GHCR publishing should use GITHUB_TOKEN with packages: write; no extra registry secret is needed.
  • The runtime server still needs production Node dependencies such as fastify, @fastify/static, and better-sqlite3. The image will contain production node_modules, but not the frontend build job's node_modules.
  • SQLite data remains external via /data and LEADS_DB_PATH.

Implementation Steps

  1. Clean repository boundaries:
    • Verify no build output remains in dist.
    • Remove local generated screenshots from the working tree if they are not needed.
    • Keep .data, node_modules, screenshots, and Playwright output ignored.
  2. Initialize git:
    • Run git init in WebSite.
    • Set branch to main.
    • Inspect git status --short before any commit.
  3. Create or connect GitHub repo:
    • Create BatchVault_Website under dzarlax if it does not exist.
    • Add remote origin.
    • Do not push until the user explicitly approves a commit/push step.
  4. Add runtime-only image build:
    • Prefer adding Dockerfile.image so the current local multi-stage Dockerfile can remain useful for manual builds.
    • Dockerfile.image will require dist/client to already exist in the Docker build context.
    • Final image will copy only dist, server, package.json, and package-lock.json, then run production install.
  5. Add GitHub Actions workflow:
    • Job 1: checkout, setup Node 22, npm ci, npm run check, npm run build, upload dist/client.
    • Job 2: checkout, download frontend artifact to dist/client, setup Buildx, login to GHCR, build and push image with Dockerfile.image.
    • Use docker/metadata-action for tags: branch, SHA, and latest on default branch.
  6. Verify locally before commit:
    • npm run check
    • npm run build
    • docker build -f Dockerfile.image -t batchvault_website:test .
    • Run container with a temp data volume and smoke /healthz plus landing page HTML if Docker is available.
    • Remove dist after local build verification to avoid committing generated output.

Proposed GitHub Actions Shape

Workflow: .github/workflows/build-image.yml

  • on: push to main and workflow_dispatch.
  • permissions: contents: read, packages: write.
  • build_frontend uploads dist/client as frontend-dist.
  • build_image depends on build_frontend, downloads frontend-dist, and builds ghcr.io/dzarlax/batchvault_website.

Risks

  • better-sqlite3 is a native dependency. If prebuilt binaries are unavailable for the GitHub runner or image platform, production install may need build tools in the Dockerfile.
  • GitHub repo name BatchVault_Website is fine for GitHub, but the image name must remain lowercase.
  • If the deploy stack later expects a different image name, personal deploy config must be updated together with the GHCR package reference.
  • If the current local screenshots are not ignored, they can accidentally enter the first commit.
  • GitHub package visibility and permissions may need a one-time UI adjustment if the image is used by a private VPS pull.

Rollback Plan

  • If the workflow fails, revert the workflow and Dockerfile changes while keeping the landing code intact.
  • If repo initialization is wrong before push, remove the local .git folder only after explicit confirmation.
  • If the GitHub repo is created with the wrong name, rename it in GitHub before publishing deploy references.
  • If GHCR image naming is wrong, stop publishing the bad package and update workflow tags before wiring deployment.
