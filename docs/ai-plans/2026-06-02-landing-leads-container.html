BatchVault landing leads and container plan

Plan date: 2026-06-02. Scope: turn the current static landing page into a small deployable website service that serves the landing page and collects contact leads without making the landing heavy.

Task Summary

Replace the current mailto:-only contact form with a normal lead collection path while keeping the landing page simple. The website should be a Node/Vite React app with a small Fastify API, containerized for deployment behind Traefik, storing leads in SQLite on a persistent volume and optionally notifying Telegram.

Current Behavior

  • WebSite/index.html is a static single-file landing page with SR/RU/EN copy and a contact form.
  • The form currently opens a prepared email to hello@batchvault.app; it does not centrally store leads.
  • BatchVault_Back is a Go/Gin/GORM/Postgres backend. Its main /api group is protected by JWT and workspace middleware after auth routes.
  • Backend migrations are run through database.ConnectDatabase() and DB.AutoMigrate(...).
  • Production deployment for the product currently lives in personal_ai_stack/deploy/jerkyvault/docker-compose.yaml with Traefik labels for frontend and backend containers.

Desired Behavior

  • The landing remains visually and technically simple: one page, three languages, no heavy frontend framework.
  • The contact form submits to a public API endpoint, receives a clear success/failure response, and does not require auth.
  • Leads are stored in a local SQLite database on a persistent Docker volume with enough context to follow up and understand source language.
  • After a successful lead insert, the service can send a Telegram notification to the configured chat.
  • The website runs as a small container and can be added to the existing Traefik deployment.
  • Lead collection has basic abuse controls: rate limiting, input validation, length limits, and a honeypot field.

Recommended Architecture

Website service

Build a Node service inside WebSite. Vite/React renders the landing page. Fastify serves the production build, exposes POST /api/leads, opens SQLite at LEADS_DB_PATH, and runs a minimal table creation/migration on startup.

Why React here

The landing already has multiple sections, three languages, form states, and future copy/design iteration. React keeps the page maintainable without making the public experience heavy. The production build remains static assets plus one small API server.

Alternative: add POST /api/leads to BatchVault_Back and store in Postgres. This is less container sprawl, but it couples public marketing traffic to the product API and main product DB. For this landing page, SQLite is the cleaner default.
Telegram notification: configure TELEGRAM_BOT_TOKEN and TELEGRAM_CHAT_ID. Notifications should be best-effort: a Telegram failure must not lose the saved lead or show a failed submission after the DB insert succeeds.

Implementation Steps

  1. Create the Vite React website in WebSite and move the landing into componentized source files.
  2. Create the Fastify API with GET /healthz and POST /api/leads. In production it also serves the Vite build.
  3. Add startup migration for landing_leads in SQLite using explicit SQL. Keep the table creation obvious and dependency-light.
  4. Add validation: required contact, max lengths, accepted language values, JSON-only body, and honeypot rejection.
  5. Add simple per-IP or per-hash rate limiting. If raw IP storage is avoided, use in-memory request rate limiting at runtime and optional salted hash only in the lead row.
  6. After the lead is inserted, send a best-effort Telegram message with name, contact, product, language, and message preview.
  7. Change the React form to submit JSON to /api/leads, then show localized success/failure text. Keep email as visible fallback copy.
  8. Add Dockerfile and hot-reload dev compose.
  9. Run local service and submit a test lead to the SQLite file.
  10. After approval for deploy work, add a website service to the deployment compose with Traefik host labels and production env vars.

Test Plan

  • SQLite verification: confirm one inserted row has expected contact, language, source, and timestamp.
  • Telegram verification: with test env vars, confirm one lead sends one readable Telegram notification. With missing/invalid Telegram env vars, confirm the lead still saves and the response remains successful.
  • Container build test: docker build for the website image.
  • Local container smoke: health endpoint returns 200 and landing loads.

Manual QA Checklist

  • Landing still opens at / and preserves the current visual style.
  • Language switcher still updates hero, pricing, and form copy.
  • Contact form can submit without reloading the page.
  • Success message is localized and does not overpromise immediate response.
  • Spam honeypot field is invisible to normal users.
  • Network failure shows a localized fallback telling the user to email directly.
  • Mobile viewport remains readable and does not horizontally scroll.

Risks

  • Spam: public forms get spam. Mitigation: honeypot, rate limit, max lengths, and no heavy captcha in the first pass.
  • Privacy: storing contact data creates responsibility. Mitigation: store only submitted fields, avoid raw IP by default, keep records easy to delete/export from SQLite.
  • Deploy drift: website container and product containers could diverge. Mitigation: keep a clear compose entry and smoke checklist.
  • SQLite persistence: losing the Docker volume would lose leads. Mitigation: explicit named volume or host bind mount plus backup/export instructions.
  • Telegram failure: Telegram can be down or misconfigured. Mitigation: save first, notify second, log notification errors without failing the form.

Rollback Plan

  • Frontend rollback: restore the form to the existing mailto: behavior.
  • Service rollback: remove the website container from compose and serve static HTML again.
  • DB rollback: keep the SQLite file for audit/export, or archive/delete it after confirming no useful leads are inside.
  • Deploy rollback: revert the Traefik route/service entry to the previous known-good compose.
